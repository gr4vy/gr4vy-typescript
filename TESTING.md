# Testing

This SDK ships a test suite that runs against the **E2E sandbox**
(`api.sandbox.e2e.gr4vy.app`). It covers the full merchant transaction lifecycle
and the back-office configuration surface, and reaches every API operation.

> **Note for contributors:** this repository is generated by
> [Speakeasy](https://www.speakeasy.com/). Everything under `tests/`,
> `vitest.config.ts`, `tsconfig.test.json`, `scripts/`, and this file is **not**
> regenerated and is safe to edit. Test dependencies are pinned in
> `.speakeasy/gen.yaml` (`additionalDependencies.devDependencies`) so they
> survive regeneration of `package.json` — add new ones there too, not just to
> `package.json`.

## Prerequisites

You need the signing key for the E2E sandbox. Provide it in either way:

- Save it as `private_key.pem` in the repo root (git-ignored), **or**
- Export it: `export PRIVATE_KEY="$(cat /path/to/private_key.pem)"`

`tests/utils/setup.ts` reads `PRIVATE_KEY` first, then falls back to
`private_key.pem`.

## Running the tests

```sh
npm install

# Everything (offline unit tests + live E2E)
npx vitest run

# A single suite, or just the headline lifecycle story
npx vitest run tests/flows
npx vitest run tests/processing/transactions.test.ts

# Watch mode while developing
npx vitest
```

The offline suites (`tests/auth.test.js`, `tests/webhook.test.ts`) need no key.

## How it's organised

```
tests/
  utils/      shared helpers — see below
  flows/      end-to-end merchant stories (transaction & buyer lifecycles)
  processing/ checkout sessions, transactions, buyers, payment methods,
              gift cards, digital wallets, payment links
  backoffice/ merchant accounts (+3DS), payment services, reports, payouts,
              audit logs, 3DS scenarios, account updater
```

### Isolation & parallelism

Every test file calls `setupMerchant()` in `beforeAll`, which provisions its
**own merchant account** (with a `mock-card` payment service) and returns a
merchant-scoped client. Because files never share a merchant account, vitest can
run them concurrently across worker threads, and CI can split them across shards
without any coordination. Within a file, tests run sequentially.

### Shared helpers (`tests/utils/`)

| File | Purpose |
| --- | --- |
| `setup.ts` | `setupMerchant()` / `setupEnvironment()`, scoped client factory, forward-compat response fetcher |
| `fixtures.ts` | Test cards, addresses, buyers, cart items, `uniqueId()`, `unwrapTransaction()` |
| `transactions.ts` | `authorizeViaCheckoutSession()` — the realistic authorise helper |
| `fields.ts` | Tokenises a card into a checkout session (the Embed `/fields` endpoint) |
| `poll.ts` | `pollUntil()` for eventually-consistent reads (capture settling, etc.) |
| `arbitraries.ts` | fast-check generators (amounts, currencies, metadata, addresses) + `fcParams()` |

### Property-based tests

We use [`fast-check`](https://fast-check.dev/) (the JS analogue of Python's
Hypothesis) for "uncertainty" coverage: amount/currency echo, metadata
round-trips, and the partial-update invariant (update a *generated* subset of
fields, assert the rest is untouched). Runs are bounded (`fcParams`, small
`numRuns`) because each case is a real network round-trip. The seed is fixed and
overridable:

```sh
FC_SEED=12345 npx vitest run tests/processing/buyers.test.ts
```

### Endpoints not reproducible on `mock-card`

Some operations need infrastructure the mock merchant doesn't have (real wallet
payloads, network-token onboarding, payout-capable connections, a gift-card
service, Loon for account updater) or are explicitly unsupported by `mock-card`
(`cancel`, `sync`, `session`, `verify`). These are still **exercised at the
request level** — the SDK serialises the request and we assert the API rejects
it for a real reason — and each is clearly commented. They are never silently
skipped, so they still count toward endpoint reach.

## Coverage

```sh
npx vitest run --coverage
node scripts/endpoint-coverage.mjs
```

Coverage is scoped to `src/funcs/**` and `src/sdk/**`. Since `src/funcs/*.ts` is
**one file per API operation**, coverage over that directory doubles as an
"endpoints reached" metric. `scripts/endpoint-coverage.mjs` turns the
`coverage-summary.json` into a Markdown report (endpoints reached + statement /
branch / function / line coverage, and a list of any operations not reached).

## CI & the release pipeline

CI (`.github/workflows/ci.yaml`) runs on every PR and push to `main`, plus a
weekday schedule:

- **`test`** — a `node-version × shard` matrix (Node 20/22/24 × 4 shards) runs
  `vitest run --shard=$i/4`. Sharding keeps each job fast while every PR stays
  fully covered. `PRIVATE_KEY` comes from a repo secret.
- **`coverage`** — runs the full suite once with `--coverage` and posts a sticky
  PR comment with the report above. It is **not** a dependency of `merge`, so it
  never blocks shipping.
- **`merge`** — for generated SDK update PRs (opened by `github-actions[bot]`),
  a green `test` job auto-approves and squash-merges the PR, cutting the next
  release. Keeping the suite fast (via sharding) is what keeps this pipeline
  flowing; keeping `coverage` off the critical path means a reporting hiccup
  never holds up a release.
